# 确定技术架构并建立工程骨架

## 任务说明

依据 Mission 与 PRD V1.7，形成可实施的系统架构，并建立供后续开发使用的工程骨架。方案需覆盖用户端 H5、PC Web 管理端、服务端、数据持久化、OpenAI 流式调用、身份与权限隔离、调用记录、初始化、测试和腾讯云部署边界，但不得扩大 MVP 范围。

## 必需交付

- 一份架构决策与模块边界说明；
- 数据模型、关键状态、接口和流式交互设计；
- 一份可机器读取的 OpenAPI 3.x 或等价 API 合同，覆盖路径、方法、认证、参数、请求体、响应体、状态码、错误码、分页和幂等语义；
- 一份流式协议合同，明确 SSE 或等价机制的事件类型、事件载荷、顺序、完成、失败、停止和断线恢复语义；
- 可供前端独立开发的请求响应示例、流式事件示例、Mock 方案和合同校验方式；
- 用户端与管理员身份隔离、会话越权防护和凭据处理方案；
- OpenAI 聊天、标题生成、重试及实际 Prompt/返回记录方案；
- 可启动的工程骨架、无真实密钥的环境变量示例和最小运行说明；
- 对后续开发 Work Item 的明确接口约束。

## 验收标准

- 覆盖 PRD P0 功能及 Mission 的部署目标；
- 技术选型明确且适合在单台腾讯云服务器部署；
- 数据结构能够表达用户、会话、消息、LLM 配置和每次 LLM 调用；
- 明确 SSE 或等价流式机制、停止生成、失败和重试的状态处理；
- API 合同覆盖用户端和管理端所需全部 P0 接口，并可通过解析器、校验器或等价方式检查；
- 前端无需读取未审核的后端 Run，即可依据已接受的合同 Artifact 和 Mock 开始开发；
- 普通用户与管理员的认证、授权和数据访问边界可检查；
- 工程骨架可在无生产凭据的本地环境完成安装或最小启动检查；
- 不包含 API Key、SSH 私钥、Token、服务器密码或其他凭据。

## API 合同协作规则

- WI-001 经 Coordinator 接受的 Artifact 是前后端并行开发的初始 API 合同基线；
- 合同尚未接受时，参与 Agent 可通过 `question_asked` 和 `question_answered` Message 澄清设计，并由 WI-001 的执行 Agent在同一 Run 中合并结论；
- 合同接受后，任何一级 Agent 不得单方面改变路径、字段、状态码、错误码、认证、分页或流式事件语义；
- 只涉及解释且不改变合同语义的问题，继续使用 `question_asked` 和 `question_answered`；
- 需要改变合同语义时，提出方必须向 Coordinator 发送 `work_item_proposed` Request，最少说明变更原因、受影响接口、兼容性、受影响 Work Items 和建议迁移方式；
- Coordinator 审核通过后，创建独立的 API 合同修订 Work Item；其交付必须是完整合并后的新合同，不得只提供零散差异；
- 新合同 Artifact 接受后，Coordinator 递增受影响 Work Items 的 revision 并明确引用新的合同 Artifact，相关 Run 才能按新合同继续；
- 若提议改变 Mission 范围、验收标准或必需交付物，由 Coordinator 请求 Mission Owner 决策，不自行批准。
